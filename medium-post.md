# 개발자의 주 업무가 코드 작성에서 Spec 리뷰로 바뀐다

## Harness Engineering — AI가 코드를 쓰는 시대, 개발자는 무엇을 해야 하는가?

2025년 하반기, 소프트웨어 개발 업계의 주요 플레이어들은 Spec을 코드보다 상위의 산출물로 다루는 방향으로 수렴하고 있다. OpenAI의 Sean Grove는 AI Engineer World's Fair에서 이를 선언했고, AWS는 Kiro라는 Spec-Driven IDE를 출시했으며, GitHub는 Spec Kit이라는 오픈소스 툴킷을 공개했다.

하지만 한 가지 빠진 이야기가 있다. 그 Spec을 **누가, 어떻게** 쓰는가? 대부분의 논의는 "개발자가 Spec을 작성하면 AI가 코드를 생성한다"에서 멈춘다. 이 글에서는 그 다음 질문을 다룬다.

> **이 글의 핵심 주장:** AI의 추측 영역을 최소화하고 코드 생성 결과를 더 예측 가능하게 만들려면, 고도로 구조화된 Spec이 필요하다. 하지만 그 Spec을 개발자가 처음부터 직접 작성하게 하면, 코드를 짜는 것보다 더 오래 걸릴 수 있다. 따라서 **개발자는 Dev Request를 쓰고, AI가 설계 문서(HLD/LLD)와 Spec을 생성하고, 개발자가 리뷰하는** 3계층 구조가 필요하다.

---

## 01. 왜 Spec의 결정성이 중요한가

AI에게 "할인 적용 후 금액에 세금을 계산해줘"라고 말하면, AI는 코드를 만들어 준다. 하지만 **어떤 코드**를 만들지는 AI가 추측한다. 할인이 소계를 초과하면? 면세 상품이 섞여 있으면? 반올림은 어느 자릿수에서?

이 추측 영역이 넓을수록 코드의 품질 편차가 커지고, 테스트의 기준도 모호해진다. 반대로 **Spec이 이 모든 것을 명시하면**, AI는 추측 없이 코드를 생성할 수 있고, 테스트도 Spec에서 자동 도출된다.

Spec 결정성에는 세 단계가 있다.

**자연어 프롬프트** — "세금 계산 기능 만들어줘" 수준. AI 추측 영역이 크다. 코드 품질 편차가 크고 재개발 시 재현이 불가하다.

**유저스토리 + 수용기준** — Kiro, Spec Kit이 제공하는 수준. AI 추측 영역이 보통이다. 어느 정도 일관성은 있지만 누락 가능성이 존재한다.

**구조화된 Spec** — 규칙 + 엣지케이스 + 참조가 명시된 수준. AI 추측 영역이 최소화된다. 높은 일관성으로 Spec 기반 재생성이 가능하다.

Harness Engineering의 핵심 전제는 바로 이것이다. 이상적으로는 **Spec이 프로젝트를 재현하는 가장 중요한 상위 설계 자산**이 되어야 하고, Spec을 기반으로 프로젝트의 주요 구조와 동작을 재생성할 수 있어야 한다. 이를 위해 Spec의 결정성은 최대한 높아야 한다.

## 02. Spec의 계층 구조

하나의 서비스를 AI가 완전히 생성하려면, "주문 생성 유스케이스" 하나만으로는 부족하다. 그 유스케이스가 참조하는 도메인 규칙, 그 규칙이 따르는 회사 정책, 그 정책이 근거하는 보안 가이드까지 계층적으로 연결되어 있어야 한다.

계층은 다음과 같다:

**🏢 전사 가이드** — 보안 정책, 인증/인가 표준, 공통 코딩 컨벤션. 변경이 가장 드물다.

**🏗️ 조직 가이드** — 기술 스택, 아키텍처 패턴, 테스트 전략, CI/CD.

**⚙️ 서비스 정의** — 프로젝트 컨텍스트, 패키지 구조, 외부 시스템 연동, 인프라.

**📦 도메인 · 정책** — 엔티티, 관계, 불변 규칙, 비즈니스 정책.

**🎬 유스케이스** — 트리거, 전제조건, 규칙, 엣지케이스. 변경이 가장 빈번하다.

이 구조의 장점은 명확하다. AI가 유스케이스 코드를 생성할 때, 상위 Spec이 하위 Spec의 맥락이 되어 AI 출력이 조직 규칙과 도메인 정책을 벗어나지 않도록 제약을 제공한다. 컨벤션 위반이나 보안 누락 같은 문제가 **구조적으로 방지**된다.

## 03. 문제: 이 Spec을 개발자가 직접 쓸 수 있는가?

여기서 많은 Spec-Driven 논의가 깊게 다루지 않은 실무 문제가 나타난다. 결정성 높은 Spec은 아래와 같은 모습이 된다:

```
## Rule: 세금 계산

### Trigger
- 주문 체크아웃 시점에 실행한다

### Preconditions
- 주문에 1개 이상의 상품이 존재해야 한다
- 주문 상태가 `PENDING` 이어야 한다

### Formula
- `tax = (subtotal - discount) * tax_rate`

### Edge Cases
| 조건                    | 결과                              |
|------------------------|----------------------------------|
| discount >= subtotal   | tax = 0                          |
| 주문에 면세 상품 포함     | 면세 상품을 subtotal에서 제외 후 계산 |

### Constraints
- tax_rate: `shipping_address.state` 기준 → 세율 정책 참조
- 반올림: `HALF_UP`, 소수점 2자리

### References
- 할인 정책 (domain/order.md#discount-policy)
- 세율 정책 (policy/tax-rates.md)
```

순수 Markdown이지만, 정해진 섹션 구조를 따르고, 엣지케이스를 테이블로 명시하며, 다른 Spec 문서를 링크로 참조한다. 이 수준의 Spec이 있어야 AI가 추측 없이 코드를 생성하고, 테스트 케이스도 자동으로 도출할 수 있다.

> ⚠️ **현실적인 문제:** 이 정도 구조의 Spec을 개발자가 모든 유스케이스에 대해 직접 작성하면, **그냥 코드를 짜는 것보다 더 오래 걸릴 수 있다.** 생산성을 높이려고 도입한 체계가 오히려 생산성을 갉아먹는 역설이 발생한다. 또한 10명의 개발자가 쓰면 10가지 스타일의 Spec이 나온다. 물론 템플릿으로 어느 정도 표준화할 수 있다. 하지만 엣지케이스 누락, 참조 문서 연결, 하위 규칙 반영 같은 작업까지 사람에게 일관되게 요구하면 결국 작성 비용이 다시 커진다. AI 입장에서 입력 포맷의 비일관성은 곧 코드 품질 저하로 이어진다.

기존 Spec-Driven 논의들(Grove, Kiro, Spec Kit)이 "개발자가 Spec을 직접 쓸 수 있다"고 말하는 이유는, 그들이 말하는 Spec의 결정성 수준이 상대적으로 낮기 때문이다. 자연어 유저스토리 + 수용 기준 정도의 수준이면 개발자가 쓸 수 있다. 하지만 **그 수준의 Spec으로는 AI의 추측 영역이 여전히 넓다.**

## 04. 해법: 3계층 문서 구조

이 딜레마의 해법은 **문서를 세 계층으로 나누는 것**이다.

**기획자 PRD** → **개발자가 Dev Request 작성** → **AI가 HLD/LLD 생성** → **개발자 설계 리뷰** → **AI가 Spec 생성** → **개발자 Spec 리뷰/수정** → **AI가 코드 + 테스트 생성**

### Layer 3: Dev Request — 개발자가 쓴다

기획자의 PRD를 기술적 관점으로 번역한 문서다. 업계에서 쓰이는 RFC(Request for Comments)나 Design Doc의 경량화 버전에 해당한다. 자연어로 쓰되, 변경 의도, 영향 범위, As-Is/To-Be, 금지 사항 같은 '판단의 입력'을 담는다. 개발 상황에 따라 무게가 달라진다:

- **신규 개발** (작성 비용: 높음, 1회성) — 도메인 모델 초안, 핵심 규칙, 기술 스택, 외부 연동
- **기능 추가** (작성 비용: 중간) — 기능 목적, 관련 도메인, 새 비즈니스 규칙
- **마이너 수정** (작성 비용: 낮음, 30초~2분) — As-Is / To-Be 한 줄씩, 관련 도메인

마이너 수정의 경우 Dev Request는 이 정도면 충분하다:

```
# DR-2025-003: 세금 계산 오류 수정

## 변경 대상
- domain: order (세금 계산 규칙)

## As-Is
- 할인 적용 전 금액에 세금 계산

## To-Be
- 할인 적용 후 금액에 세금 계산
- discount가 subtotal 초과 시 tax = 0
```

### Layer 2: HLD / LLD — AI가 생성하고, 개발자가 리뷰한다

Dev Request를 받아 AI가 생성하는 설계 문서다. HLD(High-Level Design)는 전체 아키텍처와 컴포넌트 간 관계를, LLD(Low-Level Design)는 클래스, 인터페이스, 데이터 흐름의 상세 설계를 담는다.

이 문서의 핵심 목적은 **개발자 간 커뮤니케이션**이다. 개발자들에게 익숙한 포맷이라 팀 내 공유와 리뷰가 자연스럽고, "AI가 설계를 제대로 이해했는지" 확인하는 체크포인트가 된다. 규모별로 대규모 변경은 HLD+LLD를, 중규모 기능 추가는 LLD만 거치며, 마이너 수정은 이 단계를 생략하고 바로 Spec으로 넘어간다.

### Layer 1: AI-facing Spec — Single Source of Truth

프로젝트의 유일한 진실의 원천(Single Source of Truth)이다. AI가 코드를 생성할 때 읽는 문서이자, 트리거, 전제조건, 규칙, 엣지케이스, 참조 링크 같은 '구현 가능한 구조'를 담는다.

중요한 점은, **개발자가 Spec을 직접 수정할 수 있다**는 것이다. AI가 생성한 Spec에서 빠진 엣지케이스를 추가하거나, 잘못된 참조를 바로잡는 것은 리뷰어로서 당연한 행위다. 코드 리뷰에서 버그를 발견하면 티켓을 다시 쓰지 않고 코드를 직접 고치는 것과 같다.

## 05. 각 문서의 역할과 정합성

3계층 구조에서 가장 중요한 원칙은 **정합성 체크의 범위를 제한하는 것**이다. 모든 문서 간 동기화를 강제하면 유지보수 지옥에 빠진다.

**Dev Request** — "왜 시작했는가"의 기록. 동기와 의도를 보존한다. Spec에서 참조 링크만 유지.

**HLD / LLD** — 생성 시점의 설계 스냅샷. 개발자 커뮤니케이션용. Spec이 진화해도 굳이 동기화하지 않는다.

**Spec (SSOT)** — 유일한 진실의 원천. 직접 수정 가능. 정합성 체크 대상은 Spec ↔ 코드 사이만.

> **수정 전략:** 방향이 틀렸을 때 (변경 대상 도메인 오지정, 비즈니스 요구 누락) → Dev Request를 수정하고 재생성. 세부사항이 맞지 않을 때 (엣지케이스 누락, 설계 선택 변경, 참조 오류) → Spec을 직접 수정. 매번 Dev Request로 돌아가서 재생성하면 비효율적인 루프에 빠진다. 직접 수정한 내용이 최종 Spec에 반영되면 그것이 곧 새로운 진실이다.

## 06. 왜 모든 변경이 Spec에 반영되어야 하는가

"버그 수정은 Spec 없이 바로 코드만 고치면 안 되나?"라는 질문이 나올 수 있다. 원칙적으로는 그렇지 않다. 버그 수정도 결국 프로젝트 지식의 일부이기 때문이다.

첫째, **테스트의 근거가 사라진다.** Spec이 테스트의 소스인데, Spec 없이 코드만 바꾸면 "왜 이 테스트가 이렇게 되어있는지" 추적이 안 된다. 나중에 AI가 Spec 기반으로 테스트를 재생성하면 해당 변경이 반영 안 된 테스트가 나온다.

둘째, **재개발 시 누락된다.** Harness Engineering의 궁극적 목표는 Spec 기반으로 프로젝트의 주요 구조와 동작을 재생성하는 것이다. Spec에 없는 변경은 그냥 사라진다.

이것이 3계층 구조가 필수인 이유다. 마이너 수정이라도 Dev Request는 30초면 쓸 수 있고, AI가 문서 정규화와 갱신의 기계적 작업을 맡아주므로, 개발자는 구조화 문서 작성보다 검증에 더 집중할 수 있다. 추가 부담을 0으로 만들 수는 없지만, 사람이 직접 구조화 Spec을 작성하는 비용보다는 크게 줄일 수 있다. 하지만 그 30초의 기록이 프로젝트의 지식을 보존한다.

다만 긴급 장애 대응처럼 시간 제약이 큰 경우에는 코드 선반영 후 사후 Spec 동기화를 허용할 수 있다. 핵심은 "Spec 없는 변경이 영구적으로 남지 않도록" 관리하는 것이다.

## 07. 업계는 같은 방향을 가리키고 있다

이 접근이 고립된 아이디어가 아니라는 점을 확인하기 위해, 2025년 하반기의 주요 움직임을 살펴보자.

**"The New Code" — OpenAI · Sean Grove · 2025.07**
"Spec을 쓰는 사람이 곧 프로그래머다." Spec을 living document로 관리하고, 코드 리뷰하듯 Spec 리뷰를 하라.

**Kiro IDE — AWS · 2025.07 출시**
Spec-Driven IDE. 자연어 → 유저스토리 → 설계 → 태스크의 구조화된 워크플로우를 IDE 레벨에서 지원.

**Spec Kit — GitHub · 2025.09 오픈소스**
Specify → Plan → Tasks 워크플로우. "개발자의 역할은 조향(steer), 코딩 에이전트가 작성(write)".

**Spec Coding — Red Hat · 2025.10**
인간이 "What"과 "How 가드레일"을 분리 작성. 아키텍처·보안·테스트용 Spec을 다중 조합 가능.

이들의 공통점은 **"Spec이 코드보다 상위의 산출물"**이라는 방향성이다. 하지만 차이점도 있다. 대부분은 개발자가 Spec을 직접 작성하는 것을 전제한다. Harness Engineering은 여기서 한 단계 더 나아가, **"개발자는 Spec을 작성하지 않고 리뷰한다"**는 포지션을 취한다.

기존 Spec-Driven과 Harness Engineering의 비교:

- **Spec 결정성:** 기존은 중간 (자연어 + 구조), Harness는 높음 (AI 최적화 포맷)
- **Spec 작성자:** 기존은 개발자 직접, Harness는 AI (개발자 입력 기반)
- **개발자 역할:** 기존은 Spec 작성 + 리뷰, Harness는 Dev Request 작성 + Spec 리뷰/수정
- **설계 공유:** 기존은 Spec 자체 또는 별도 문서, Harness는 HLD/LLD (AI 생성, 커뮤니케이션용 스냅샷)
- **AI 추측 영역:** 기존은 보통, Harness는 최소화
- **재개발 가능성:** 기존은 Spec 품질에 따라 다름, Harness는 Spec 중심 재생성 가능성 강화 (목표)
- **정합성 체크:** 기존은 명시되지 않음, Harness는 Spec ↔ 코드 사이만 (문서 간 동기화 최소화)

## 08. 개발자의 역할은 어떻게 바뀌는가

Harness Engineering에서 개발자의 핵심 역량은 세 가지로 수렴한다.

**첫째, 기술적 번역 능력.** 기획자의 비즈니스 요구를 기술적 맥락으로 번역하는 것. "이 기능을 추가하면 기존 주문 도메인의 세금 계산에 영향이 간다"를 판단하고 Dev Request에 명시하는 능력이다.

**둘째, Spec 검증 능력.** AI가 생성한 Spec에서 빠진 엣지케이스를 찾고, 잘못된 참조를 잡아내고, 비즈니스 규칙의 모순을 발견하는 능력이다. 이것은 코드 리뷰와 본질적으로 같은 역량인데, 대상이 코드에서 Spec으로 옮겨진 것이다.

**셋째, 시스템 설계 능력.** Spec의 계층 구조를 설계하고, 도메인 간 경계를 정하고, 기술 스택과 아키텍처 결정을 내리는 것. 이것은 AI가 대체할 수 없는 영역이다.

## 마치며

"Spec이 새로운 코드다"라는 흐름은 이미 업계의 주요 플레이어들 사이에서 점점 선명해지고 있다. 하지만 **그 Spec을 누가 쓰는가**에 대한 답은 아직 충분히 논의되지 않았다.

Harness Engineering의 제안은 간명하다. AI의 추측 영역을 최소화하려면 높은 결정성의 Spec이 필요하고, 그 Spec을 개발자가 직접 쓰게 하면 생산성이 떨어지므로, **개발자는 "무엇을 만들어야 하는지"를 쓰고 (Dev Request), AI는 "어떻게 설계하고 구현해야 하는지"를 쓰고 (HLD/LLD → Spec), 개발자는 "이게 맞는지"를 검증하고 직접 수정한다 (Spec Review)**.

코드 리뷰가 구현 결과를 사후 검증하는 활동이라면, Spec 리뷰는 구현 이전에 의도와 제약을 고정함으로써 오류 비용을 더 앞단에서 줄이는 활동이다. 개발자의 전문성이 가장 효과적으로 발휘되는 지점에 집중하는 것. 그것이 Harness Engineering이 지향하는 개발자 역할의 미래다.
