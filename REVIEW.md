## 수정 사항 (우선순위 순)

### 1. 서두 ↔ 07장 중복 정리
- 도입부에서 OpenAI/AWS/GitHub를 이미 나열하고, 07장에서 거의 동일 내용 반복
- 서두는 업계 흐름 1-2문장 요약으로 줄이고, 07장에서 각 플레이어의 접근법 차이를 **비교 분석** 형태로 재구성

### 2. 04장에 DR→Spec 변환 예시 추가
- 세금 계산 예시를 활용하여 Dev Request → (HLD/LLD) → Spec으로 변환되는 구체적 과정 보여주기
- 이 글의 가장 큰 약점이 "구체적 예시의 부재"

### 3. "Harness Engineering" 정의 추가
- 첫 등장 시 "이 글에서는 이러한 접근을 Harness Engineering이라 부른다" 명시

### 4. 05+06장 통합
- 분량이 짧은 05(정합성)와 06(버그 수정)을 "Spec 유지보수와 정합성 관리"로 통합

### 5. NFR + 구현 힌트 섹션 추가 (새로운 내용)
- 핵심 논점: 코드 레벨에서만 구현 최적화를 처리하면 나중에 재현 불가 → 결정성 떨어짐
- NFR 섹션에 성능 기준 + "구현 힌트" 남기는 방식 제안
- 예시: NFR에 "응답 시간 p95 < 200ms" 기준 + 구현 힌트로 "이 테이블은 파티셔닝 안 하면 풀스캔 발생" 같은 시행착오 지식 기록
- 이것이 Spec의 결정성을 높이고 재개발 시 지식 유실을 방지한다는 논리

### 6. 한계/반론 섹션 추가
- 프로토타이핑/탐색적 개발에는 안 맞을 수 있음
- 레거시 시스템 적용의 어려움
- AI가 생성한 Spec의 품질 검증 문제
- 한계를 인정하면 오히려 신뢰도 상승

### 7. 표현/용어 개선
- 제목: "대체" → "코드보다 먼저인 시대" 또는 유사 표현으로 수정
- "수렴하고 있다" → "움직이고 있다" 또는 "실험하고 있다"
- "역설" → "딜레마" (트레이드오프이므로)
- HLD/LLD/PRD 약어 첫 등장 시 풀어쓰기
- "바로 이것이다" 같은 구어적 표현 정돈
- "프로젝트를 재생성" → "주요 비즈니스 로직과 도메인 규칙을 재생성"으로 범위 한정
- "코드보다 오래 걸린다" 부분에 구체적 시간 비교 추가

### 8. 04장에 "Dev Request만으로 AI가 완벽한 Spec을 만드는 건 논리적 모순" 해소
- "AI는 상위 계층 Spec을 컨텍스트로 참조하여 빈틈을 채운다. 누락 시 개발자에게 역질문" 한 줄 추가

### 9. 08장(결론) 보강
- 현재 너무 일반적. 구체적으로 어떤 스킬을 키워야 하는지, 필자의 관점(낙관/우려) 담기
