Harness Engineering

개발자의 주 업무가
코드 작성에서
Spec 리뷰로 바뀐다

AI가 코드를 쓰는 시대, 개발자는 무엇을 해야 하는가?
Spec-Driven Development의 이상과 현실 사이에서 찾은 답.

2025년 하반기, 소프트웨어 개발 업계에서 하나의 합의가 형성되고 있다. "Spec이 새로운 코드다." OpenAI의 Sean Grove는 AI Engineer World's Fair에서 이를 선언했고, AWS는 Kiro라는 Spec-Driven IDE를 출시했으며, GitHub는 Spec Kit이라는 오픈소스 툴킷을 공개했다.

하지만 한 가지 빠진 이야기가 있다. 그 Spec을 누가, 어떻게 쓰는가? 대부분의 논의는 "개발자가 Spec을 작성하면 AI가 코드를 생성한다"에서 멈춘다. 이 글에서는 그 다음 질문을 다룬다.

💡 이 글의 핵심 주장

AI가 결정적(deterministic)으로 코드를 생성하려면 고도로 구조화된 Spec이 필요하다. 하지만 그 Spec을 개발자가 처음부터 직접 작성하게 하면, 코드를 짜는 것보다 더 오래 걸릴 수 있다. 따라서 개발자는 '개발 요청서'를 쓰고, AI가 Spec을 생성하고, 개발자가 리뷰하는 2계층 구조가 필요하다.

01

왜 Spec의 결정성이 중요한가

AI에게 "할인 적용 후 금액에 세금을 계산해줘"라고 말하면, AI는 코드를 만들어 준다. 하지만 어떤 코드를 만들지는 AI가 추측한다. 할인이 소계를 초과하면? 면세 상품이 섞여 있으면? 반올림은 어느 자릿수에서?

이 추측 영역이 넓을수록 코드의 품질 편차가 커지고, 테스트의 기준도 모호해진다. 반대로 Spec이 이 모든 것을 명시하면, AI는 추측 없이 코드를 생성할 수 있고, 테스트도 Spec에서 자동 도출된다.

결정성이 높을수록 AI의 추측 영역이 줄고, 코드 품질이 안정된다

Harness Engineering의 핵심 전제는 바로 이것이다. Spec이 프로젝트의 Single Source of Truth가 되어야 하고, 그 Spec만 있으면 프로젝트를 처음부터 다시 만들 수 있어야 한다. 이를 위해 Spec의 결정성은 최대한 높아야 한다.

02

Spec의 계층 구조

하나의 서비스를 AI가 완전히 생성하려면, "주문 생성 유스케이스" 하나만으로는 부족하다. 그 유스케이스가 참조하는 도메인 규칙, 그 규칙이 따르는 회사 정책, 그 정책이 근거하는 보안 가이드까지 계층적으로 연결되어 있어야 한다.

상위 계층은 변경이 드물고, 하위로 갈수록 빈번하게 갱신된다

이 구조의 장점은 명확하다. AI가 유스케이스 코드를 생성할 때, 도메인 규칙을 참조하고, 조직의 아키텍처 패턴을 따르며, 전사 보안 정책을 준수하는 코드를 만들 수 있다. 컨벤션 위반이나 보안 누락 같은 문제가 구조적으로 방지된다.

03

문제: 이 Spec을 개발자가 직접 쓸 수 있는가?

여기서 기존 Spec-Driven Development 논의들이 간과하는 지점이 나온다. 결정성 높은 Spec은 아래와 같은 모습이 된다: 

## Rule: 세금 계산

### Trigger
- 주문 체크아웃 시점에 실행한다

### Preconditions
- 주문에 1개 이상의 상품이 존재해야 한다
- 주문 상태가 `PENDING` 이어야 한다

### Formula
- `tax = (subtotal - discount) * tax_rate`

### Edge Cases

| 조건                    | 결과                              |
|------------------------|----------------------------------|
| discount >= subtotal   | tax = 0                          |
| 주문에 면세 상품 포함     | 면세 상품을 subtotal에서 제외 후 계산 |

### Constraints
- tax_rate: `shipping_address.state` 기준 → [세율 정책](../policy/tax-rates.md)
- 반올림: `HALF_UP`, 소수점 2자리

### References
- [할인 정책](../domain/order.md#discount-policy)
- [세율 정책](../policy/tax-rates.md)

순수 Markdown이지만, 정해진 섹션 구조를 따르고, 엣지케이스를 테이블로 명시하며, 다른 Spec 문서를 링크로 참조한다. 이 수준의 Spec이 있어야 AI가 추측 없이 코드를 생성하고, 테스트 케이스도 자동으로 도출할 수 있다.

⚠️ 현실적인 문제

이 정도 구조의 Spec을 개발자가 모든 유스케이스에 대해 직접 작성하면, 그냥 코드를 짜는 것보다 더 오래 걸릴 수 있다. 생산성을 높이려고 도입한 체계가 오히려 생산성을 갉아먹는 역설이 발생한다.

또한 10명의 개발자가 쓰면 10가지 스타일의 Spec이 나온다. AI 입장에서 입력 포맷의 비일관성은 곧 코드 품질 저하로 이어진다.

기존 Spec-Driven 논의들(Grove, Kiro, Spec Kit)이 "개발자가 Spec을 직접 쓸 수 있다"고 말하는 이유는, 그들이 말하는 Spec의 결정성 수준이 상대적으로 낮기 때문이다. 자연어 유저스토리 + 수용 기준 정도의 수준이면 개발자가 쓸 수 있다. 하지만 그 수준의 Spec으로는 AI의 추측 영역이 여전히 넓다.

04

해법: 2계층 문서 구조

이 딜레마의 해법은 문서를 두 계층으로 나누는 것이다.

개발자는 "쓰기 쉬운 문서"를 쓰고, AI는 "결정성 높은 문서"를 만든다

Layer 2: 개발 요청서 (Dev Request) — 개발자가 쓴다

기획자의 PRD를 기술적 관점으로 번역한 문서. 자연어로 쓰되, AI가 Spec을 생성하는 데 필요한 핵심 정보만 담는다. 개발 상황에 따라 무게가 달라진다:

상황 Dev Request 작성 비용 포함 내용
신규 개발 높음 (1회성) 도메인 모델 초안, 핵심 규칙, 기술 스택, 외부 연동
기능 추가 중간 기능 목적, 관련 도메인, 새 비즈니스 규칙
마이너 수정 낮음 (30초~2분) As-Is / To-Be 한 줄씩, 관련 도메인
대규모 마이그레이션 높음 As-Is/To-Be 상태, 변환 규칙, 호환성 요구사항

마이너 수정의 경우 개발 요청서는 이 정도면 충분하다: 

# DR-2025-003: 세금 계산 오류 수정

## 변경 대상
- domain: order (세금 계산 규칙)

## As-Is
- 할인 적용 전 금액에 세금 계산

## To-Be
- 할인 적용 후 금액에 세금 계산
- discount가 subtotal 초과 시 tax = 0

이걸 AI가 받아서, 기존 Spec 문서를 분석하고, 섹션 3에서 본 것 같은 구조화된 Spec으로 변환한다. 개발자는 그 결과를 리뷰하면 된다.

Layer 1: AI-facing Spec — AI가 생성하고 관리한다

프로젝트의 Single Source of Truth. AI가 코드를 생성할 때 읽는 문서이자, 프로젝트를 처음부터 재개발할 때의 유일한 입력. 개발자가 직접 편집하지 않고, Dev Request를 통해서만 변경된다.

05

왜 모든 변경이 Spec에 반영되어야 하는가

"버그 수정은 Spec 없이 바로 코드만 고치면 안 되나?"라는 질문이 나올 수 있다. 답은 안 된다이다. 두 가지 이유가 있다.

첫째, 테스트의 근거가 사라진다. Spec이 테스트의 소스인데, Spec 없이 코드만 바꾸면 "왜 이 테스트가 이렇게 되어있는지" 추적이 안 된다. 나중에 AI가 Spec 기반으로 테스트를 재생성하면 해당 변경이 반영 안 된 테스트가 나온다.

둘째, 재개발 시 누락된다. Harness Engineering의 궁극적 목표는 "Spec만으로 프로젝트를 재현"하는 것이다. Spec에 없는 변경은 그냥 사라진다.

모든 변경은 Spec을 거쳐야 한다. 그래야 Dev Request 비용이 30초여도 괜찮다.

이것이 2계층 구조가 필수인 이유다. 마이너 수정이라도 Dev Request는 30초면 쓸 수 있고, AI가 Spec을 갱신해주므로 개발자의 추가 부담은 거의 없다. 하지만 그 30초의 기록이 프로젝트의 지식을 보존한다.

06

이들의 공통점은 "Spec이 코드보다 상위의 산출물"이라는 합의다. 하지만 차이점도 있다. 대부분은 개발자가 Spec을 직접 작성하는 것을 전제한다. Harness Engineering은 여기서 한 단계 더 나아가, "개발자는 Spec을 작성하지 않고 리뷰한다"는 포지션을 취한다.

기존 Spec-Driven Harness Engineering
Spec 결정성 중간 (자연어 + 구조) 높음 (AI 최적화 포맷)
Spec 작성자 개발자 직접 AI (개발자 입력 기반)
개발자 역할 Spec 작성 + 리뷰 Dev Request 작성 + Spec 리뷰
AI 추측 영역 보통 최소화
재개발 가능성 Spec 품질에 따라 다름 Spec만으로 재현 가능 (목표)
07

개발자의 역할은 어떻게 바뀌는가

개발자의 가치는 "타이핑 속도"에서 "판단의 정확도"로 이동한다

Harness Engineering에서 개발자의 핵심 역량은 세 가지로 수렴한다.

첫째, 기술적 번역 능력. 기획자의 비즈니스 요구를 기술적 맥락으로 번역하는 것. "이 기능을 추가하면 기존 주문 도메인의 세금 계산에 영향이 간다"를 판단하고 Dev Request에 명시하는 능력이다.

둘째, Spec 검증 능력. AI가 생성한 Spec에서 빠진 엣지케이스를 찾고, 잘못된 참조를 잡아내고, 비즈니스 규칙의 모순을 발견하는 능력이다. 이것은 코드 리뷰와 본질적으로 같은 역량인데, 대상이 코드에서 Spec으로 옮겨진 것이다.

셋째, 시스템 설계 능력. Spec의 계층 구조를 설계하고, 도메인 간 경계를 정하고, 기술 스택과 아키텍처 결정을 내리는 것. 이것은 AI가 대체할 수 없는 영역이다.

마치며

"Spec이 새로운 코드다"라는 선언은 이미 업계의 합의로 향하고 있다. 하지만 그 Spec을 누가 쓰는가에 대한 답은 아직 충분히 논의되지 않았다.

Harness Engineering의 제안은 간명하다. AI가 결정적으로 코드를 생성하려면 높은 결정성의 Spec이 필요하고, 그 Spec을 개발자가 직접 쓰게 하면 생산성이 떨어지므로, 개발자는 "무엇을 만들어야 하는지"를 쓰고 (Dev Request), AI는 "어떻게 만들어야 하는지"를 쓰고 (Spec), 개발자는 "이게 맞는지"를 검증한다 (Spec Review).

코드 리뷰가 코드 작성보다 레버리지가 높듯이, Spec 리뷰는 Spec 작성보다 레버리지가 높다. 개발자의 전문성이 가장 효과적으로 발휘되는 지점에 집중하는 것. 그것이 Harness Engineering이 지향하는 개발자 역할의 미래다.

이 글은 실제 Spec-Driven Development 실험과 업계 동향 분석을 기반으로 작성되었습니다.
