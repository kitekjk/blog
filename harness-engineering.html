Harness Engineering

개발자의 주 업무가
코드 작성에서
Spec 리뷰로 바뀐다

Spec이 코드보다 먼저인 시대, 개발자는 무엇을 하는가?
Spec-Driven Development의 실무적 해법을 제안한다.

2025년 하반기, 소프트웨어 업계의 주요 플레이어들이 Spec을 코드보다 상위의 산출물로 다루는 방향으로 움직이고 있다. 하지만 한 가지 빠진 이야기가 있다. 그 Spec을 누가, 어떻게 쓰는가?

대부분의 논의는 "개발자가 Spec을 작성하면 AI가 코드를 생성한다"에서 멈춘다. 이 글에서는 그 다음 질문을 다룬다.

이 글의 핵심 주장

AI의 추측 영역을 최소화하고 코드 생성 결과를 더 예측 가능하게 만들려면, 고도로 구조화된 Spec이 필요하다. 하지만 그 Spec을 개발자가 처음부터 직접 작성하게 하면, 코드를 짜는 것보다 더 오래 걸릴 수 있다. 따라서 개발자는 Dev Request를 쓰고, AI가 설계 문서(HLD/LLD)와 Spec을 생성하고, 개발자가 리뷰하는 3계층 구조가 필요하다.

01

왜 Spec의 결정성이 중요한가

AI에게 "할인 적용 후 금액에 세금을 계산해줘"라고 말하면, AI는 코드를 만들어 준다. 하지만 어떤 코드를 만들지는 AI가 추측한다. 할인이 소계를 초과하면? 면세 상품이 섞여 있으면? 반올림은 어느 자릿수에서?

이 추측 영역이 넓을수록 코드의 품질 편차가 커지고, 테스트의 기준도 모호해진다. 반대로 Spec이 이 모든 것을 명시하면, AI는 추측 없이 코드를 생성할 수 있고, 테스트도 Spec에서 자동 도출된다. 이 글에서는 이러한 접근을 Harness Engineering이라 부른다.

결정성이 높을수록 AI의 추측 영역이 줄고, 코드 품질 편차가 줄어든다

Spec 결정성에는 세 단계가 있다. 자연어 프롬프트는 "세금 계산 기능 만들어줘" 수준으로, AI 추측 영역이 크다. 유저스토리 + 수용기준은 Kiro, Spec Kit이 제공하는 수준으로, 어느 정도 일관성은 있지만 누락 가능성이 존재한다. 구조화된 Spec은 규칙, 엣지케이스, 참조가 명시된 수준으로, AI 추측 영역이 최소화된다.

Harness Engineering의 핵심 전제는 이 전제 위에 성립한다. 이상적으로는 Spec이 프로젝트를 재현하는 가장 중요한 상위 설계 자산이 되어야 하고, Spec을 기반으로 주요 비즈니스 로직과 도메인 규칙을 재생성할 수 있어야 한다. 이를 위해 Spec의 결정성은 최대한 높아야 한다.

02

Spec의 계층 구조

하나의 서비스를 AI가 완전히 생성하려면, "주문 생성 유스케이스" 하나만으로는 부족하다. 그 유스케이스가 참조하는 도메인 규칙, 그 규칙이 따르는 회사 정책, 그 정책이 근거하는 보안 가이드까지 계층적으로 연결되어 있어야 한다.

상위 계층은 변경이 드물고, 하위로 갈수록 빈번하게 갱신된다

이 구조의 장점은 명확하다. AI가 유스케이스 코드를 생성할 때, 상위 Spec이 하위 Spec의 맥락이 되어, AI 출력이 조직 규칙과 도메인 정책을 벗어나지 않도록 제약을 제공한다. 컨벤션 위반이나 보안 누락 같은 문제가 구조적으로 방지된다.

03

문제: 이 Spec을 개발자가 직접 쓸 수 있는가?

여기서 많은 Spec-Driven 논의가 깊게 다루지 않은 실무 문제가 나타난다. 결정성 높은 Spec은 아래와 같은 모습이 된다: 

## Rule: 세금 계산

### Trigger
- 주문 체크아웃 시점에 실행한다

### Preconditions
- 주문에 1개 이상의 상품이 존재해야 한다
- 주문 상태가 `PENDING` 이어야 한다

### Formula
- `tax = (subtotal - discount) * tax_rate`

### Edge Cases
| 조건                    | 결과                              |
|------------------------|----------------------------------|
| discount >= subtotal   | tax = 0                          |
| 주문에 면세 상품 포함     | 면세 상품을 subtotal에서 제외 후 계산 |

### Constraints
- tax_rate: `shipping_address.state` 기준 → 세율 정책 참조
- 반올림: `HALF_UP`, 소수점 2자리

### References
- 할인 정책 (domain/order.md#discount-policy)
- 세율 정책 (policy/tax-rates.md)

순수 Markdown이지만, 정해진 섹션 구조를 따르고, 엣지케이스를 테이블로 명시하며, 다른 Spec 문서를 링크로 참조한다. 이 수준의 Spec이 있어야 AI가 추측 없이 코드를 생성하고, 테스트 케이스도 자동으로 도출할 수 있다.

⚠️ 현실적인 딜레마

이 정도 구조의 Spec을 개발자가 모든 유스케이스에 대해 직접 작성하면, 그냥 코드를 짜는 것보다 더 오래 걸릴 수 있다. 구조화된 Spec 작성에 2–3시간, 코드 직접 작성에 1시간이라면, 생산성을 높이려고 도입한 체계가 오히려 생산성을 갉아먹는 딜레마가 발생한다.

또한 10명의 개발자가 쓰면 10가지 스타일의 Spec이 나온다. 물론 템플릿으로 어느 정도 표준화할 수 있다. 하지만 엣지케이스 누락, 참조 문서 연결, 하위 규칙 반영 같은 작업까지 사람에게 일관되게 요구하면 결국 작성 비용이 다시 커진다. AI 입장에서 입력 포맷의 비일관성은 곧 코드 품질 저하로 이어진다.

기존 Spec-Driven 논의들(Grove, Kiro, Spec Kit)이 "개발자가 Spec을 직접 쓸 수 있다"고 말하는 이유는, 그들이 말하는 Spec의 결정성 수준이 상대적으로 낮기 때문이다. 자연어 유저스토리 + 수용 기준 정도의 수준이면 개발자가 쓸 수 있다. 하지만 그 수준의 Spec으로는 AI의 추측 영역이 여전히 넓다.

04

해법: 3계층 문서 구조

이 딜레마의 해법은 문서를 세 계층으로 나누는 것이다.

개발자는 쓰기 쉬운 문서를 쓰고, AI가 결정성 높은 문서를 만든다

Layer 3: Dev Request — 개발자가 쓴다

기획자의 PRD(Product Requirements Document)를 기술적 관점으로 번역한 문서다. 업계에서 쓰이는 RFC(Request for Comments)나 Design Doc의 경량화 버전에 해당한다. 자연어로 쓰되, 변경 의도, 영향 범위, As-Is/To-Be, 금지 사항 같은 '판단의 입력'을 담는다. 개발 상황에 따라 무게가 달라진다:

신규 개발 (작성 비용: 높음, 1회성) — 도메인 모델 초안, 핵심 규칙, 기술 스택, 외부 연동. 기능 추가 (작성 비용: 중간) — 기능 목적, 관련 도메인, 새 비즈니스 규칙. 마이너 수정 (작성 비용: 낮음, 30초~2분) — As-Is / To-Be 한 줄씩, 관련 도메인.

마이너 수정의 경우 Dev Request는 이 정도면 충분하다:

# DR-2025-003: 세금 계산 오류 수정

## 변경 대상
- domain: order (세금 계산 규칙)

## As-Is
- 할인 적용 전 금액에 세금 계산

## To-Be
- 할인 적용 후 금액에 세금 계산
- discount가 subtotal 초과 시 tax = 0

이 Dev Request를 받은 AI는 기존 상위 Spec(도메인 규칙, 세율 정책 등)을 컨텍스트로 참조하여 다음과 같은 Spec을 생성한다: 

## Rule: 세금 계산 (updated via DR-2025-003)

### Trigger
- 주문 체크아웃 시점에 실행한다

### Formula
- `tax = (subtotal - discount) * tax_rate`

### Edge Cases
| 조건                    | 결과                              |
|------------------------|----------------------------------|
| discount >= subtotal   | tax = 0                          |
| 주문에 면세 상품 포함     | 면세 상품을 subtotal에서 제외 후 계산 |

### Constraints
- tax_rate: `shipping_address.state` 기준 → 세율 정책 참조
- 반올림: `HALF_UP`, 소수점 2자리

Dev Request 자체는 간결하지만, AI는 상위 계층 Spec을 컨텍스트로 참조하여 빈틈을 채운다. 도메인 규칙, 세율 정책, 할인 정책 같은 기존 Spec이 AI가 구조화된 Spec을 생성하는 데 필요한 맥락을 제공한다. 만약 참조할 상위 Spec이 부족하면, AI는 개발자에게 역질문한다.

Layer 2: HLD(High-Level Design) / LLD(Low-Level Design) — AI가 생성하고, 개발자가 리뷰한다

Dev Request를 받아 AI가 생성하는 설계 문서다. HLD는 전체 아키텍처와 컴포넌트 간 관계를, LLD는 클래스, 인터페이스, 데이터 흐름의 상세 설계를 담는다.

이 문서의 핵심 목적은 개발자 간 커뮤니케이션이다. 개발자들에게 익숙한 포맷이라 팀 내 공유와 리뷰가 자연스럽고, "AI가 설계를 제대로 이해했는지" 확인하는 체크포인트가 된다. 규모별로 대규모 변경은 HLD+LLD를, 중규모 기능 추가는 LLD만 거치며, 마이너 수정은 이 단계를 생략하고 바로 Spec으로 넘어간다.

Layer 1: AI-facing Spec — Single Source of Truth

프로젝트의 유일한 진실의 원천(Single Source of Truth)이다. AI가 코드를 생성할 때 읽는 문서이자, 트리거, 전제조건, 규칙, 엣지케이스, 참조 링크 같은 '구현 가능한 구조'를 담는다.

중요한 점은, 개발자가 Spec을 직접 수정할 수 있다는 것이다. AI가 생성한 Spec에서 빠진 엣지케이스를 추가하거나, 잘못된 참조를 바로잡는 것은 리뷰어로서 당연한 행위다. 코드 리뷰에서 버그를 발견하면 티켓을 다시 쓰지 않고 코드를 직접 고치는 것과 같다.

05

Spec 유지보수와 정합성 관리

3계층 구조에서 가장 중요한 원칙은 정합성 체크의 범위를 제한하는 것이다. 모든 문서 간 동기화를 강제하면 유지보수 지옥에 빠진다.

Layer 3
Dev Request

"왜 시작했는가"의 기록. 동기와 의도를 보존한다. Spec에서 참조 링크만 유지.

Layer 2
HLD / LLD

생성 시점의 설계 스냅샷. 개발자 커뮤니케이션용. Spec이 진화해도 굳이 동기화하지 않는다.

Layer 1 · SSOT
Spec

유일한 진실의 원천. 직접 수정 가능. 정합성 체크 대상은 Spec ↔ 코드 사이만.

수정 전략

방향이 틀렸을 때 (변경 대상 도메인 오지정, 비즈니스 요구 누락) → Dev Request를 수정하고 재생성.

세부사항이 맞지 않을 때 (엣지케이스 누락, 설계 선택 변경, 참조 오류) → Spec을 직접 수정.

매번 Dev Request로 돌아가서 재생성하면 비효율적인 루프에 빠진다. 직접 수정한 내용이 최종 Spec에 반영되면 그것이 곧 새로운 진실이다.

왜 모든 변경이 Spec에 반영되어야 하는가

"버그 수정은 Spec 없이 바로 코드만 고치면 안 되나?"라는 질문이 나올 수 있다. 원칙적으로는 그렇지 않다. 버그 수정도 결국 프로젝트 지식의 일부이기 때문이다.

모든 변경은 Spec을 거쳐야 한다. Dev Request 비용이 30초여도 괜찮다.

첫째, 테스트의 근거가 사라진다. Spec이 테스트의 소스인데, Spec 없이 코드만 바꾸면 "왜 이 테스트가 이렇게 되어있는지" 추적이 안 된다. 나중에 AI가 Spec 기반으로 테스트를 재생성하면 해당 변경이 반영 안 된 테스트가 나온다.

둘째, 재개발 시 누락된다. Harness Engineering의 궁극적 목표는 Spec 기반으로 주요 비즈니스 로직과 도메인 규칙을 재생성하는 것이다. Spec에 없는 변경은 그냥 사라진다.

이것이 3계층 구조가 필수인 이유다. 마이너 수정이라도 Dev Request는 30초면 쓸 수 있고, AI가 문서 정규화와 갱신의 기계적 작업을 맡아주므로, 개발자는 구조화 문서 작성보다 검증에 더 집중할 수 있다. 추가 부담을 0으로 만들 수는 없지만, 사람이 직접 구조화 Spec을 작성하는 비용보다는 크게 줄일 수 있다. 하지만 그 30초의 기록이 프로젝트의 지식을 보존한다.

긴급 대응 예외

긴급 장애 대응처럼 시간 제약이 큰 경우에는 코드 선반영 후 사후 Spec 동기화를 허용할 수 있다. 핵심은 "Spec 없는 변경이 영구적으로 남지 않도록" 관리하는 것이다.

06

NFR과 구현 힌트

Spec이 "무엇을(What)" 정의하고 코드가 "어떻게(How)" 구현한다는 원칙은 명확하다. 하지만 구현 레벨의 최적화를 코드에서만 처리하면, 재개발 시 그 최적화를 재현할 수 없다. 쿼리 튜닝, 인덱스 설계, 캐시 전략 같은 시행착오로 얻은 지식이 코드 안에만 묻혀 있으면, AI가 Spec 기반으로 코드를 재생성할 때 같은 시행착오를 처음부터 반복하게 된다.

⚠️ Spec에 Why와 What만 있고 How가 없으면, AI는 같은 시행착오를 반복한다

성능 기준은 NFR로 명시할 수 있지만, 그 기준을 달성하기 위해 거친 시행착오까지 Spec에 남기지 않으면 결정성은 떨어질 수밖에 없다.

이를 해결하기 위해 Spec의 NFR(비기능 요구사항) 섹션에 성능 기준과 함께 구현 힌트(Implementation Hints)를 남긴다. 

### NFR: 성능
- 주문 조회 API: p95 < 200ms, p99 < 500ms

### 구현 힌트 (Implementation Hints)
- orders 테이블: created_at 기준 월별 파티셔닝 필수 (미적용 시 풀스캔)
- 세금 계산: tax_rate 조회를 매 요청마다 하지 않고 캐시 적용 (TTL 1h)
- 할인 검증: discount_rules 테이블 JOIN 대신 선계산된 discount_amount 컬럼 사용

구현 힌트는 Spec의 결정성을 높이지 않지만, 시행착오로 얻은 운영 지식을 보존한다. AI가 코드를 재생성할 때 이 힌트를 참조하면, "파티셔닝 없이 배포 → 풀스캔 발생 → 장애 → 파티셔닝 적용"이라는 비용이 큰 학습 루프를 건너뛸 수 있다.

구현 힌트는 Spec의 정규 규칙(Rule, Edge Case, Constraint)과 성격이 다르다. 규칙은 "반드시 이렇게 해야 한다"는 제약이고, 힌트는 "이렇게 하면 더 낫다"는 권장이다. AI는 힌트를 무시할 수도 있지만, 대부분의 경우 따르는 것이 합리적이다.

07

이 접근이 고립된 아이디어가 아니라는 점을 확인하기 위해, 2025년 하반기의 주요 움직임을 살펴보자.

인간이 "What"과 "How 가드레일"을 분리 작성. 아키텍처·보안·테스트용 Spec을 다중 조합 가능.

이들의 공통점은 "Spec이 코드보다 상위의 산출물"이라는 방향성이다. 하지만 차이점도 있다. 대부분은 개발자가 Spec을 직접 작성하는 것을 전제한다. Harness Engineering은 여기서 한 단계 더 나아가, "개발자는 Spec을 작성하지 않고 리뷰한다"는 포지션을 취한다.

기존 Spec-Driven Harness Engineering
Spec 결정성 중간 (자연어 + 구조) 높음 (AI 최적화 포맷)
Spec 작성자 개발자 직접 AI (개발자 입력 기반)
개발자 역할 Spec 작성 + 리뷰 Dev Request 작성 + Spec 리뷰/수정
설계 공유 Spec 자체 또는 별도 문서 HLD/LLD (AI 생성, 커뮤니케이션용 스냅샷)
AI 추측 영역 보통 최소화
재개발 가능성 Spec 품질에 따라 다름 Spec 중심 재생성 가능성 강화 (목표)
정합성 체크 명시되지 않음 Spec ↔ 코드 사이만 (문서 간 동기화 최소화)
08

개발자의 역할은 어떻게 바뀌는가

개발자의 가치는 "타이핑 속도"에서 "판단의 정확도"로 이동한다

Harness Engineering에서 개발자의 핵심 역량은 세 가지로 수렴한다.

첫째, 기술적 번역 능력. 기획자의 비즈니스 요구를 기술적 맥락으로 번역하는 것. "이 기능을 추가하면 기존 주문 도메인의 세금 계산에 영향이 간다"를 판단하고 Dev Request에 명시하는 능력이다.

둘째, Spec 검증 능력. AI가 생성한 Spec에서 빠진 엣지케이스를 찾고, 잘못된 참조를 잡아내고, 비즈니스 규칙의 모순을 발견하는 능력이다. 이것은 코드 리뷰와 본질적으로 같은 역량인데, 대상이 코드에서 Spec으로 옮겨진 것이다.

셋째, 시스템 설계 능력. Spec의 계층 구조를 설계하고, 도메인 간 경계를 정하고, 기술 스택과 아키텍처 결정을 내리는 것. 이것은 AI가 대체할 수 없는 영역이다.

마치며

"Spec이 새로운 코드다"라는 흐름은 이미 업계의 주요 플레이어들 사이에서 점점 선명해지고 있다. 하지만 그 Spec을 누가 쓰는가에 대한 답은 아직 충분히 논의되지 않았다.

Harness Engineering의 제안은 간명하다. AI의 추측 영역을 최소화하려면 높은 결정성의 Spec이 필요하고, 그 Spec을 개발자가 직접 쓰게 하면 생산성이 떨어지므로, 개발자는 "무엇을 만들어야 하는지"를 쓰고 (Dev Request), AI는 "어떻게 설계하고 구현해야 하는지"를 쓰고 (HLD/LLD → Spec), 개발자는 "이게 맞는지"를 검증하고 직접 수정한다 (Spec Review).

물론 프로토타이핑이나 탐색적 개발처럼 Spec이 오히려 방해가 되는 단계도 있다. 레거시 시스템에 도입하는 것도 쉽지 않다. AI가 생성한 Spec의 품질을 검증하는 것 자체가 새로운 역량을 요구하기도 한다. 하지만 이러한 한계가 있더라도, "코드만 남기고 설계 의도는 버리는" 기존 방식보다는 Spec을 중심에 두는 접근이 더 나은 출발점이라고 본다.

코드 리뷰가 구현 결과를 사후 검증하는 활동이라면, Spec 리뷰는 구현 이전에 의도와 제약을 고정함으로써 오류 비용을 더 앞단에서 줄이는 활동이다. 개발자의 전문성이 가장 효과적으로 발휘되는 지점에 집중하는 것. 그것이 Harness Engineering이 지향하는 개발자 역할의 미래다.

이 글은 실제 Spec-Driven Development 실험과 업계 동향 분석을 기반으로 작성되었습니다.
